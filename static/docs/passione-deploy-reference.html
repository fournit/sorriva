Deploy & Schema Reference
Canonical schemas and path reference for roadmap-data.json, docs-manifest.json, web-assets.json, deploy-manifest.json, and deploy zip construction. Use this at the start of any session that touches these files.
This doc is the schema source of truth. The session protocol covers workflow and process. This doc covers exact field names, types, required/optional status, file locations, and zip paths. When in doubt about a field, check here first.

Section 1
System file map
Manifests are complete lists, never additive. docs-manifest.json and web-assets.json define exactly what the Hub renders — an entry missing from the manifest means it doesn't appear in the panel, even if the file is on disk. Always read the current manifest from the repo package before modifying it.

System JSON files are invisible in the Hub UI. docs-manifest.json, web-assets.json, roadmap-data.json, and deploy-manifest.json do not appear in the Docs or Web Assets panels. They can only be inspected via terminal (cat) or by downloading the repo package. Always pull a fresh repo package at session start when working on any of these files.

Pre-packaging verification — mandatory before every zip
Before running the zip command, verify every filename against the file map in Section 1. The most common error is prefixing system files with the product name. Use this checklist:

Section 3
roadmap-data.json schema
Top-level structure
{ "phases": [ { "phase": 1, integer — phase number "name": "Foundation", display name for this phase "desc": "Optional subtitle", shown in Timeline view "features": [ …feature objects… ] } ] }
phases format is canonical. The Hub also supports a legacy flat array format (features with a "phase" string field) but all new roadmaps must use the phases structure above. Never write a flat array for a new product.
Feature object — full schema
{ "id": "fCamelCase", fCamelCase — unique across all phases "name": "Human name", display name shown in all roadmap views "desc": "Description", optional — shown in feature detail modal "priority": "Normal", High | Normal | Low "complexity": "M", XS | S | M | L | XL "tag": "feature", feature | bug | enhancement | debt "done": false, boolean — true when complete "priority_rank": 1, integer 1–N, null if not ranked or done "createdAt": "YYYY-MM-DD", date added — drives the New filter in List view "completedAt": "YYYY-MM-DD", set when done:true — null or omit when pending "requires": ["fOtherId"] array of blocking feature IDs — hard deps only }
FieldNotes
idRequiredfCamelCase format. Auto-generated by Hub modal. Must be unique across all phases in the file.
nameRequiredDisplay name. Used in all views and in sessions.json features_shipped array.
descOptionalShown in feature detail modal only. Keep to 1–2 sentences.
priorityRequiredHigh · Normal · Low. Drives priority filter in List view.
complexityRequiredXS=3h · S=8h · M=20h · L=40h · XL=80h equivalent dev hours.
tagRequiredfeature · bug · enhancement · debt. Bugs are excluded from roadmap progress % calculation.
doneRequiredfalse when pending, true when complete. Set completedAt and null priority_rank at the same time.
priority_rankOptionalInteger 1–N. 1 = highest priority. null for unranked features and all done features. Set by Claude at session close per priority rules.
createdAtRequiredYYYY-MM-DD. Set to today when adding a feature. Drives the New filter — features without this field never appear under New.
completedAtOptionalYYYY-MM-DD. Set when marking done:true. Shown as a badge in List view Done tab.
requiresOptionalArray of feature IDs this feature is blocked by. Hard blockers only — not soft dependencies. Hub renders a blocked indicator and dependency chain in detail modal.
New filter — how createdAt is used

The List view New filter shows features whose createdAt (or legacy added field) falls within the selected lookback window. The filter uses a day-offset from midnight, not a 24-hour rolling window.

Lookback optionCutoff logicShows
Todaycutoff = today at 00:00:00Features added today only
Last 2 days (default)cutoff = 2 days ago at 00:00:00Today + yesterday + day before
Last 7 dayscutoff = 7 days ago at 00:00:00Features added in the past week
Last 30 dayscutoff = 30 days ago at 00:00:00Features added in the past month
Always set createdAt when adding features manually to roadmap-data.json. Features without createdAt are invisible to the New filter regardless of when they were added.
Session close rules for roadmap

At session close, Claude must:

# For completed features: "done": true "completedAt": "YYYY-MM-DD" today's date "priority_rank": null always null done features # For newly added features: "createdAt": "YYYY-MM-DD" today's date — never omit # priority_rank reassignment rules (in order): # 1. Blocking other features → highest rank # 2. XS/S quick win closing a gap → high # 3. Current active phase → normal minimum # 4. Future phase, no current deps → low / null
Section 4
deploy-manifest.json schema

Lives at root/ inside the deploy zip → extracts to {APP_PATH}/deploy-manifest.json. Hub reads it on extraction to pre-fill the git commit message field. Required in session-close zips. Optional in iteration zips.

{ "message": "v0.6.1 — Plain English summary of what this deploy contains", "version": "0.6.1", "db_migrate": false }
FieldNotes
messageRequiredGit commit message. Format: "v{version} — {plain English summary}". Written by Claude at session close. Hub pre-fills this into the commit message field — omitting it means Tom has to type it manually. Survives page reloads via sessionStorage.
versionRequiredVersion string matching version.json exactly.
db_migrateRequiredtrue if this deploy requires a DB schema migration. Hub shows a warning and migrate button in post-deploy actions. false otherwise.
Section 5
docs-manifest.json schema

Lives at {APP_PATH}/static/docs/docs-manifest.json. Deploy zip path: static/docs/docs-manifest.json. The Docs panel renders exactly what this file contains — nothing more, nothing less.

{ "product": "tack", "docs": [ { "file": "tack-handoff.html", "display_name": "Handoff", "description": "Current state and session continuity context.", "category": "handoff" } ] }
FieldNotes
productRequiredProduct key string. Matches the product key used in zip naming.
fileRequiredFilename only — no path prefix. File must exist in static/docs/ on disk. Any file type supported (.html, .json, .pdf, .md).
display_nameRequiredHuman-readable name shown in the Docs panel.
descriptionOptionalShort description shown below display_name in the panel.
categoryRequiredControls panel grouping. See categories below.
Valid categories
ValueUse for
handoffProduct handoff docs, session continuity context
archArchitecture, infrastructure, session protocol, deploy guides
designDesign systems, UI specs, brand guidelines
marketingPositioning, messaging, ICP, competitive docs
testingTest plans, smoke tests, QA checklists
apiAPI references, OpenAPI specs, integration docs
otherAnything that doesn't fit above
Always deploy doc files and docs-manifest.json together in the same zip. A file on disk without a manifest entry is invisible in the Docs panel. A manifest entry pointing to a missing file shows a broken entry. The Hub Docs panel has an Audit button that flags both conditions.
System files are excluded from audit. docs-manifest.json, web-assets.json, roadmap-data.json, deploy-manifest.json, and *.meta.json sidecars are never flagged as orphans by the audit tool.
Section 6
web-assets.json schema

Lives at {APP_PATH}/static/docs/web-assets.json. Deploy zip path: static/docs/web-assets.json. Supports two entry types: file assets and URL links. Both coexist in the same assets array.

File asset entry
{ "file": "mockup-v2.html", "label": "Feature Mockup", "description": "Optional", "type": "mockup", "added": "YYYY-MM-DD" }
URL link entry
{ "url": "https://...", "label": "External Resource", "description": "Optional", "type": "reference", "added": "YYYY-MM-DD" }
FieldApplies toNotes
fileRequired*File entriesFilename only — no path. File must exist in static/web/ on disk. *Required for file entries, omit for URL entries.
urlRequired*URL entriesFull URL including https://. *Required for URL entries, omit for file entries.
labelRequiredBothDisplay name shown in the Web Assets panel.
descriptionOptionalBothShort description. Shown in the panel below the label.
typeRequiredBothmockup · prototype · marketing · reference. Controls filter bar and color coding.
addedRequiredBothYYYY-MM-DD. Date the asset was added. Set to today when adding manually.
Always deploy asset files and web-assets.json together in the same zip. File entries without the actual file on disk will 404 on preview. Deploying the manifest without the file produces broken entries in the panel.
file is a filename, not a path. Do not include subdirectory segments — web/my-file.html is wrong, my-file.html is correct. The Hub always serves files from static/web/ regardless of what is written in the file field.

Sorriva is artifact-only. No Flask backend, no launchctl daemon, no server restart. Deploy zips never contain app.py for Sorriva. Roadmap, docs, and web assets deploy exactly as for any other product.
Passione Hub frontend file is hub.html, not passione.html. Every other product follows the {product-key}.html convention (tack.html, sorriva.html, etc.). Hub is the exception — app.py hardcodes send_from_directory("static", "hub.html"). Deploy zips for Passione Hub must use static/hub.html as the zip path, not static/passione.html. A zip containing static/passione.html will extract successfully but the file will never be served.
