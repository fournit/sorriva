Session Protocol
Cross-product conventions for every Claude session — how to start, how to deploy, how to close. Read this at the start of every session alongside the product-specific handoff doc.
⚡ Session start — Claude read this first and confirm before proceeding
Timestamps — always CDT (UTC-5, Mar–Oct) or CST (UTC-6, Nov–Feb). Compute with: datetime.now(timezone.utc) - timedelta(hours=5). Never use raw UTC.
Deploy zip naming — {product}-deploy-YYYY-MM-DD-HHMMSS.zip. Run TZ='America/Chicago' date '+%Y-%m-%d-%H%M%S' immediately before packaging — no user input required, no approximation. One zip per iteration. Never reuse a filename. See product key table in Section 2 for exact key values.
Zip folder structure — root/ for app.py, sessions.json, version.json, deploy-manifest.json · static/ for HTML files · static/docs/ for roadmap, handoff, docs.
No stamped backup files — canonical files only. No app-2026-06-29-120000.py copies. Deployment zips in deployments/ serve as the rollback mechanism.
No emojis in UI — use Tabler Icons for all UI iconography. Log messages and status text may use ✓ ✗ ⚠ sparingly.
Feature IDs — always use fCamelCase roadmap IDs (e.g. fOuraAuth) in all discussions, sessions.json, and dep chains.
sessions.json duration_hrs — equivalent traditional dev hours (XS=3h, S=8h, M=20h, L=40h, XL=80h). NOT wall-clock time. Sum complexity of features shipped.
Session open — when Tom says "open/start/begin a session" (or similar), immediately ask: "What time is it?" Record the response as session_start_time. Do this before any work begins.
Session close — when closing, first state: "You started this session at [session_start_time]." Then ask: "How many hours did you personally spend?" Record as owner_hrs. Never estimate. If start time was not recorded, ask Tom to recall it.
New files — whenever a new file is generated, flag it and ask Tom: "Should this be included in the repo package?" If no, add it to .repoignore. Never silently create files that affect repo package contents.
No code changes without confirmation — Claude proposes all changes first and waits for explicit approval before modifying any file. Never write or modify code speculatively.
Terminal work — one command at a time — when guiding server or terminal work, give exactly one command, then wait for output before proceeding. Never batch multiple commands in a single response. Never embed comments (# like this) inline with commands — they cause shell errors and clutter output.
File delivery — scp pattern — Claude creates files and presents them for download. Tom scps to server: scp ~/Downloads/{file} tomfo@100.126.165.63:~/. Never use inline sudo tee or heredoc to write files directly on server. LaunchDaemon plists must be root:wheel after scp — run sudo chown root:wheel /Library/LaunchDaemons/{plist} before bootstrap.

Section 2
Development iteration workflow
Every iteration produces one deploy zip. No individual files are delivered separately. The zip is the only output from Claude. This applies to every file type without exception — code, docs, roadmap JSON, HTML documents, renamed files. A doc rename, a roadmap update, or a single HTML file is never presented as an individual download. If it needs to go to the server, it goes in a zip. Hotfixes are the only exception and are handled case by case.
Claude modifies files and packages {product}-deploy-YYYY-MM-DD-HHMMSS.zip with CDT timestamp computed at packaging time. Before packaging: (1) confirm passione-deploy-reference.html has been read this session — if not, read it now before proceeding; (2) extract the JS block from any modified .html file and run node --check. Do not package if errors are found.

Timestamp computation (Claude must use this)
Compute at package time, not at code-writing time. Building code at 11:00 and packaging at 11:30 means the zip gets 11:30, not 11:00. Never approximate or use the time Tom stated as session start. Run the command at the moment of packaging.

Claude runs this bash command immediately before creating the zip — no user input required:

Use the output directly as the zip filename suffix. CDT (UTC-5) applies March–October, CST (UTC-6) applies November–February — the TZ variable handles this automatically.

Session close workflow
Triggered by "close the session", "wrap up", or similar. Do not close without completing all steps.
State the session start time — "You started this session at [session_start_time]." If start time was not recorded, ask Tom to recall it before proceeding.

Update sessions.json — append session entry with all required fields (see sessions.json spec below).
Update version.json — bump version number if warranted, update notes.
Update product handoff doc — reflect new state, resolved questions, open issues.
Produce final deploy zip — includes all changed files plus root/deploy-manifest.json. deploy-manifest.json must always include all three required fields: message (git commit text, format: "v{version} — {plain English summary}"), version (matching version.json), and db_migrate (true/false). Hub pre-fills the commit message from the manifest — omitting it means Tom has to type it manually.
Tom deploys the final zip → hub pre-fills commit message from manifest → Tom clicks Save to GitHub.
Priority reassessment rules
  • priority_rank is the single source of truth — stored on each feature (1–N, null = not a priority). Claude sets it at session end. Completed features automatically have null rank.
  • Criteria in order: (a) blocking other features → highest priority; (b) XS/S quick win that closes a gap → high; (c) current active phase → normal minimum; (d) future phase, no current deps → low.
Section 4
Deploy zip spec
Folder structure
root/ # → {APP_PATH}/ app.py # Flask backend (triggers restart if included) version.json sessions.json deploy-manifest.json # Always include in final/session-close zips static/ # → {APP_PATH}/static/ {product}.html # Main frontend SPA static/docs/ # → {APP_PATH}/static/docs/ roadmap-data.json {product}-handoff.html tack-deployment-guide.html # or equivalent per product
Extraction rules
  • Files are extracted to canonical names — no stamped copies saved
  • If app.py is in the zip, server restarts after extraction and page auto-reloads
  • Zip is saved to {APP_PATH}/deployments/ (last 3 kept) for rollback
  • Known routing folders (root, static, docs) are never stripped as a common prefix
Allowed extensions
.py .json .html .htm .css .js .sh .txt .md .plist .png .jpg .jpeg .gif .svg .webp
Web assets convention
Always deploy asset files and web-assets.json together in the same zip. Deploying files without updating the manifest means the Web panel won't know they exist. Deploying a manifest without the files means the panel shows assets that 404 on preview.
  • Asset files — go in static/web/ inside the deploy zip → extracts to {APP_PATH}/static/web/
  • Manifest — web-assets.json goes in static/docs/ inside the deploy zip → extracts to {APP_PATH}/static/docs/web-assets.json
  • Repo package — web-assets.json is included in the repo package automatically (lives in static/docs/). Always pull a fresh repo package at session start when working on web assets.
  • Design docs — go in static/docs/ via the Docs panel, not static/web/. Do not add design docs to web-assets.json.
Docs convention
Always deploy doc files and docs-manifest.json together in the same zip. The Docs panel is manifest-driven — a file on disk without a manifest entry will not appear in the panel.
  • Doc files — go in static/docs/ inside the deploy zip. Any file type is supported (HTML, JSON, PDF, etc.).
  • Manifest — docs-manifest.json goes in static/docs/. Must be updated whenever a doc is added or removed.
  • Repo package — docs-manifest.json is included in the repo package automatically. Always pull a fresh repo package at session start when working on docs.
  • Upload via Hub UI — the Docs panel has a + Upload doc button. Uploading through the UI automatically updates docs-manifest.json on the server — no session needed for simple uploads.
  • Edit metadata via Hub UI — each doc row has an Edit button to update display name, description, and category directly in the browser. Writes to docs-manifest.json immediately.
  • Audit button — compares manifest against disk. Flags orphaned files (on disk, not in manifest) and missing files (in manifest, not on disk).
  • Every product needs its own docs-manifest.json — without it the Docs panel shows empty state.
docs-manifest.json — full schema

The manifest lives at {APP_PATH}/static/docs/docs-manifest.json.

{ "product": "tack", "docs": [ { "file": "tack-handoff.html", # Filename only — must exist in static/docs/ "display_name": "Handoff", # Name shown in the Docs panel "description": "Current state and...", # Optional. Shown below display name "category": "handoff" # Controls grouping — see categories below } ] }
FieldRequiredNotes
fileYesFilename only — no path. Must exist in static/docs/. Any file type.
display_nameYesHuman-readable name shown in the panel.
descriptionNoShort description shown below the display name.
categoryYeshandoff · arch · design · marketing · testing · api · other. Controls panel grouping.
System files excluded from audit: docs-manifest.json, web-assets.json, roadmap-data.json, deploy-manifest.json, and *.meta.json sidecars are never flagged as orphans.
web-assets.json — full schema

The manifest lives at {APP_PATH}/static/docs/web-assets.json. It is a JSON object with a single top-level key assets — an array of asset entries.

{ "assets": [ { "label": "My Feature Mockup", # Display name — groups versions together "file": "my-mockup-v2.html", # Filename only — no path. Must exist in static/web/ "type": "mockup", # One of: mockup | prototype | marketing | reference "version": 2, # Integer. Used for sorting (desc) and grouping "latest": true, # true on the newest version, false on older ones "live_url": "https://example.com" # Optional. Adds Live button linking to production URL } ] }
FieldRequiredTypeNotes
labelYesstringHuman-readable name. Entries sharing the same label are grouped as versions of the same asset.
fileYesstringFilename only — no subdirectory path. File must be deployed to static/web/.
typeYesstringmockup · prototype · marketing · reference. Controls filter bar and color coding in the Web panel.
versionYesintegerNumeric version. Higher = newer. Used for sort order within a label group.
latestYesbooleantrue on the current version, false on all older versions. Older versions are collapsed by default in the panel.
live_urlNostringFull URL to production. When present, a green Live button appears. Omit for non-public assets.
file is a filename, not a path. Do not include subdirectory segments (e.g. web/my-file.html is wrong — use my-file.html). The hub always serves files from static/web/ regardless of what is written here.
Multiple versions of the same asset — give them the same label, different version integers, and set latest: true on only one. The panel groups them and collapses older versions.

Rollback & recovery
Rollback via hub (preferred — between commits)

Use when a deployed iteration breaks something and you haven't committed yet.

Open hub → Select product → Deploy panel
Scroll to Rollback section — shows last 3 deployment zips with timestamps and sizes
Click ↺ Redeploy on the target zip — inline confirm row appears
Click "Yes, redeploy" — files are re-extracted, server restarts if app.py was in that zip
Git revert via hub (preferred — after commit)

Use when you've committed bad code to GitHub.

Open hub → Select product → Deploy panel → Commit History
Click ↩ on the commit to revert
Hub runs git revert {hash} --no-edit, amends with "Revert: {original message}", pushes to GitHub
Revert commits show orange in the history log
Terminal fallback (hub is down)
# Restart hub sudo launchctl stop system/com.passione.hub && sudo launchctl start system/com.passione.hub # Restart any product daemon sudo launchctl kickstart -k system/com.passione.{product} # Git revert manually cd /var/app/{product} sudo git revert HEAD --no-edit sudo git push origin main
Hub not reachable
# Check if hub is running curl http://localhost:5200/api/health # Check logs sudo tail -20 /var/app/passione-hub/logs/server.log # Restart hub sudo launchctl stop system/com.passione.hub && sudo launchctl start system/com.passione.hub
Product server not responding (502 / timeout)

Use when a product URL returns 502 or times out. kickstart -k works from both SSH and GUI session. launchctl load/unload only works from a GUI session on the Mac Mini itself.

# Restart product daemon sudo launchctl kickstart -k system/com.passione.{product} # If that fails, load the service first sudo launchctl load /Library/LaunchDaemons/com.passione.{product}.plist
Cloudflare tunnel down (Error 1033)

Use the Cloudflare Tunnel card in the Hub Ops panel, or from terminal:

pkill cloudflared sleep 2 cloudflared tunnel run {product} &
Webhook not firing
# Check GitHub → repo Settings → Webhooks → Recent Deliveries for errors # Verify GITHUB_WEBHOOK_SECRET in /var/app/{product}/.env matches GitHub webhook settings # Check deploy log sudo tail -20 /var/app/{product}/logs/deploy.log # Verify server is reachable curl https://{product-domain}/api/health
Git pull failing (credential error)

The PAT is embedded in the remote URL. If it expires, regenerate at github.com/settings/tokens and update:

cd /var/app/{product} git remote set-url origin https://<NEW_PAT>@github.com/{org}/{repo}.git

Also update GITHUB_TOKEN in /var/app/{product}/.env. Never commit .env to git — it is in .gitignore. Rotate both the remote URL PAT and the env var if a token is compromised.

Repo package download

The Deploy panel has a "Download repo package" button that zips all project files (app.py, HTML, docs, roadmap, sessions.json) for dropping into a new Claude session. Use this at session start to give Claude the latest codebase.

  • Flat structure — all files land at the zip root with no subfolders, regardless of whether they came from app_path/, static/, or static/docs/.
  • Exclusions via .repoignore — place a .repoignore file at the product's app root to exclude specific files by filename (one per line). Lines starting with # are comments. Files already excluded by default: logs/, deployments/, __pycache__/, .git/, hidden files, and timestamped files.
  • New file protocol — whenever a new file is created during a session, Claude must ask whether it should be included in the repo package. If no, add it to .repoignore.
  • Deploy zip vs repo package — these are different formats. Deploy zips use root/, static/, static/docs/ folder structure for hub extraction routing. Repo packages are flat for Claude context use. Never confuse the two.
